## 1. 平台概述

### 1.1 平台定位
APIJSON低代码平台是一个可视化、无代码的数据查询和业务流程编排平台，通过拖拽式界面和APIJSON语法，让非技术人员也能轻松构建复杂的数据查询和业务流程。

### 1.2 核心特性
- 🎨 **可视化查询构建** - 拖拽式界面，无需编写代码
- 🔌 **多数据源支持** - MySQL、PostgreSQL、MongoDB、Redis等
- 🔐 **完善的权限控制** - 表级、字段级、行级权限
- 🔄 **工作流编排** - 可视化设计复杂业务流程
- 📊 **实时监控** - 查询性能、系统状态、告警通知
- 🧩 **插件扩展** - 支持自定义功能扩展
- 📚 **丰富的模板** - 预置查询模板，快速复用

### 1.3 适用场景
- 📊 **数据分析** - 快速构建数据分析查询
- 📈 **报表生成** - 自动化报表数据获取
- 🔍 **数据探索** - 交互式数据查询和探索
- 🤖 **业务流程** - 自动化复杂业务流程
- 📱 **移动端开发** - 为移动端提供数据API

## 3. APIJSON查询语法

### 3.1 基础查询结构

#### 3.1.1 最简单的查询
```json
{
  "User": {}
}
```
**说明**: 查询User表的所有数据，默认返回前10条记录。

#### 3.1.2 指定查询字段
```json
{
  "User": {
    "columns": ["id", "username", "email", "createdAt"]
  }
}
```
**说明**: 只查询指定的字段，减少数据传输量。

#### 3.1.3 WHERE条件查询
```json
{
  "User": {
    "columns": ["id", "username"],
    "where": {
      "status": "active",
      "age": { "$gt": 18 }
    }
  }
}
```
**说明**: 查询状态为active且年龄大于18的用户。

### 3.2 条件操作符

| 操作符 | 说明 | 示例 |
|--------|------|------|
| `$gt` | 大于 | `{"age": {"$gt": 18}}` |
| `$gte` | 大于等于 | `{"score": {"$gte": 60}}` |
| `$lt` | 小于 | `{"age": {"$lt": 65}}` |
| `$lte` | 小于等于 | `{"score": {"$lte": 100}}` |
| `$eq` | 等于 | `{"status": {"$eq": "active"}}` |
| `$ne` | 不等于 | `{"status": {"$ne": "deleted"}}` |
| `$in` | 在列表中 | `{"id": {"$in": [1, 2, 3]}}` |
| `$nin` | 不在列表中 | `{"id": {"$nin": [4, 5, 6]}}` |
| `$like` | 模糊匹配 | `{"name": {"$like": "%张%"}}` |
| `$ilike` | 忽略大小写模糊匹配 | `{"name": {"$ilike": "%zhang%"}}` |

### 3.3 逻辑操作符

#### 3.3.1 AND条件
```json
{
  "User": {
    "where": {
      "status": "active",
      "age": { "$gt": 18 }
    }
  }
}
```
**说明**: 多个条件默认为AND关系。

#### 3.3.2 OR条件
```json
{
  "User": {
    "where": {
      "$or": [
        {"status": "active"},
        {"status": "pending"}
      ]
    }
  }
}
```
**说明**: 查询状态为active或pending的用户。

#### 3.3.3 NOT条件
```json
{
  "User": {
    "where": {
      "$not": {
        "status": "deleted"
      }
    }
  }
}
```
**说明**: 查询状态不是deleted的用户。

### 3.4 JOIN查询

#### 3.4.1 INNER JOIN
```json
{
  "User": {
    "columns": ["u.id", "u.username", "p.bio"],
    "joins": [
      {
        "type": "INNER",
        "table": "Profile",
        "alias": "p",
        "on": "u.id = p.userId"
      }
    ]
  }
}
```
**说明**: 内连接User和Profile表。

#### 3.4.2 LEFT JOIN
```json
{
  "User": {
    "joins": [
      {
        "type": "LEFT",
        "table": "Order",
        "alias": "o",
        "on": "u.id = o.userId"
      }
    ]
  }
}
```
**说明**: 左连接，即使没有订单也会返回用户信息。

#### 3.4.3 多表JOIN
```json
{
  "User": {
    "joins": [
      {
        "type": "INNER",
        "table": "Profile",
        "alias": "p",
        "on": "u.id = p.userId"
      },
      {
        "type": "LEFT",
        "table": "Order",
        "alias": "o",
        "on": "u.id = o.userId"
      }
    ]
  }
}
```

### 3.5 分组与聚合

#### 3.5.1 GROUP BY
```json
{
  "User": {
    "columns": ["department", "COUNT(*) as userCount"],
    "groupBy": ["department"]
  }
}
```
**说明**: 按部门分组统计用户数。

#### 3.5.2 HAVING条件
```json
{
  "User": {
    "groupBy": ["department"],
    "having": {
      "userCount": { "$gt": 10 }
    }
  }
}
```
**说明**: 筛选用户数大于10的部门。

### 3.6 排序与分页

#### 3.6.1 ORDER BY
```json
{
  "User": {
    "orderBy": ["createdAt-", "username+"]
  }
}
```
**说明**: 按创建时间降序、用户名升序排序。`+`表示升序，`-`表示降序。

#### 3.6.2 LIMIT和OFFSET
```json
{
  "User": {
    "limit": 20,
    "offset": 0
  }
}
```
**说明**: 返回前20条记录，offset用于分页。

### 3.7 全局指令

| 指令 | 说明 | 示例 |
|--------|------|------|
| `@method` | HTTP方法 | `"@method": "GET"` |
| `@page` | 页码 | `"@page": 1` |
| `@limit` | 每页数量 | `"@limit": 20` |
| `@offset` | 偏移量 | `"@offset": 0` |
| `@order` | 排序 | `"@order": "createdAt-"` |
| `@cache` | 缓存时间(毫秒) | `"@cache": 60000` |
| `@total` | 返回总数 | `"@total": true` |
| `@count` | 返回当前页数量 | `"@count": true` |

## 4. 数据源管理

### 4.1 数据源类型

| 类型 | 说明 | 适用场景 |
|------|------|----------|
| MySQL | 关系型数据库 | 事务、复杂查询 |
| PostgreSQL | 关系型数据库 | JSON支持、高级特性 |
| MongoDB | 文档型数据库 | 灵活schema、大数据 |
| Redis | 内存数据库 | 缓存、会话 |
| HTTP API | 外部数据源 | 第三方服务集成 |

### 4.2 数据源配置

#### 4.2.1 MySQL配置示例
```json
{
  "name": "生产数据库",
  "type": "mysql",
  "connection": {
    "host": "db.example.com",
    "port": 3306,
    "database": "production_db",
    "username": "readonly_user",
    "password": "encrypted_password",
    "ssl": true,
    "timezone": "Asia/Shanghai",
    "charset": "utf8mb4"
  },
  "options": {
    "poolSize": 10,
    "timeout": 30000,
    "maxRetries": 3,
    "connectionLimit": 100
  }
}
```

#### 4.2.2 PostgreSQL配置示例
```json
{
  "name": "PostgreSQL主库",
  "type": "postgresql",
  "connection": {
    "host": "pg.example.com",
    "port": 5432,
    "database": "app_db",
    "username": "app_user",
    "password": "password",
    "ssl": true
  },
  "options": {
    "poolSize": 20,
    "statementTimeout": 30000
  }
}
```

### 4.3 数据源操作

#### 4.3.1 创建数据源
1. 进入"数据源管理"页面
2. 点击"新建数据源"按钮
3. 选择数据源类型（MySQL/PostgreSQL/MongoDB等）
4. 填写连接信息
5. 点击"测试连接"验证配置
6. 保存数据源

#### 4.3.2 测试连接
- **连接测试**: 验证网络连通性
- **认证测试**: 验证用户名密码
- **权限测试**: 验证数据库访问权限
- **性能测试**: 测量连接延迟

#### 4.3.3 查看表结构
连接成功后，系统自动获取并展示：
- 表列表
- 字段信息
- 索引信息
- 关系图

### 4.4 最佳实践

✅ **使用只读账号**: 生产环境使用只读账号
✅ **连接池配置**: 根据并发量调整连接池大小
✅ **SSL加密**: 生产环境启用SSL
✅ **超时设置**: 设置合理的查询超时时间
✅ **定期测试**: 定期测试数据源连接状态

## 5. 用户权限系统

### 5.1 权限模型

```
用户(User)
  ↓ 拥有
角色(Role)
  ↓ 继承
权限(Permission)
  ↓ 作用于
资源(Resource)
  ↓ 限制范围
数据权限(DataPermission)
```

### 5.2 预置角色

| 角色 | 描述 | 权限范围 |
|------|------|----------|
| 超级管理员 | 拥有所有权限 | `*` |
| 管理员 | 数据源和用户管理 | `datasource:*`, `user:*` |
| 开发者 | 查询和模板管理 | `query:*`, `template:*` |
| 分析师 | 只读查询权限 | `query:read`, `query:execute` |
| 查看者 | 只读查看权限 | `query:read` |

### 5.3 权限粒度

#### 5.3.1 表级权限
```json
{
  "resource": "datasource",
  "scope": "table",
  "tables": ["User", "Order"],
  "action": "read"
}
```
**说明**: 只允许读取User和Order表。

#### 5.3.2 字段级权限
```json
{
  "resource": "datasource",
  "scope": "column",
  "table": "User",
  "columns": ["id", "username"],
  "action": "read"
}
```
**说明**: 只允许读取User表的id和username字段。

#### 5.3.3 行级权限
```json
{
  "resource": "datasource",
  "scope": "row",
  "table": "Order",
  "condition": {
    "userId": "${currentUser.id}",
    "department": "${currentUser.department}"
  },
  "action": "read"
}
```
**说明**: 只能查看自己部门的订单。

### 5.4 权限检查流程

```
用户请求
    ↓
1. 检查Token有效性
    ↓
2. 检查用户状态
    ↓
3. 加载用户角色和权限
    ↓
4. 检查资源权限
    ├─ 表级权限检查
    ├─ 字段级权限检查
    └─ 行级权限检查
    ↓
5. 允许或拒绝请求
```

### 6.2 操作步骤

#### 6.2.1 创建简单查询
1. **选择表**: 从表选择器中选择要查询的表
2. **选择字段**: 勾选需要返回的字段
3. **添加条件**: 点击"+"添加条件，设置字段和操作符
4. **设置排序**: 选择排序字段和方向
5. **预览SQL**: 查看生成的SQL语句
6. **执行查询**: 点击"执行"按钮查看结果

#### 6.2.2 创建复杂查询
1. **添加多个表**: 选择多个表进行关联查询
2. **配置JOIN**: 设置JOIN类型和关联条件
3. **添加分组**: 选择分组字段
4. **添加聚合**: 在条件中使用聚合函数
5. **保存模板**: 将复杂查询保存为模板
6. **快速复用**: 下次直接加载模板使用

### 6.3 查询模板

#### 6.3.1 保存模板
1. 构建好查询后，点击"保存模板"
2. 输入模板名称和描述
3. 选择分类（用户管理、订单管理等）
4. 设置可见性（公开/私有）
5. 保存成功后可在模板列表中查看

#### 6.3.2 使用模板
1. 进入"查询模板"页面
2. 按分类或关键词搜索模板
3. 点击"使用"按钮加载模板
4. 根据实际需求修改查询条件
5. 执行查询

## 7. 工作流编排

### 7.1 工作流节点类型

| 节点类型 | 图标 | 说明 | 配置项 |
|----------|------|------|--------|
| 数据查询 | 📊 | 数据源、查询配置 |
| 数据转换 | 🔄 | 转换脚本、映射规则 |
| 条件分支 | 🔀 | 条件表达式、分支路径 |
| 循环迭代 | 🔄 | 循环变量、退出条件 |
| API调用 | 🔗 | API地址、请求方法、参数映射 |
| 通知 | 📧 | 通知方式、接收人、消息模板 |
| 延迟 | ⏱️ | 延迟时间 |
| 结束 | 🏁 | 返回值、结束状态 |

### 7.2 工作流示例

#### 7.2.1 订单处理流程
```
开始
  ↓
[查询订单信息] → [检查库存] → [调用支付] → [更新订单状态]
                      ↓                    ↓
                  [库存不足] ← [支付失败]
                      ↓                    ↓
                  [发送补货通知] ← [发送失败通知]
```

**节点配置**:
- **查询订单信息**: 
  - 数据源: 生产数据库
  - 查询: `SELECT * FROM Order WHERE id = ${orderId}`
  
- **检查库存**:
  - 条件: `${order.stock} > 0`
  
- **调用支付**:
  - API: `https://payment.example.com/api/pay`
  - 方法: POST
  - 参数: `{orderId, amount}`

#### 7.2.2 数据同步流程
```
开始
  ↓
[查询源数据] → [数据转换] → [写入目标表] → [验证结果]
                              ↓
                        [失败重试] ← [发送告警]
```

### 7.3 工作流执行

#### 7.3.1 同步执行
- **适用场景**: 快速流程、用户交互操作
- **特点**: 等待执行完成，实时返回结果
- **超时**: 可配置超时时间

#### 7.3.2 异步执行
- **适用场景**: 长时间流程、定时任务
- **特点**: 立即返回执行ID，后台执行
- **回调**: 完成后通过Webhook通知
- **监控**: 可通过执行ID查询执行状态

## 8. 监控与告警

#### 8.1.3 业务指标

### 8.2 告警配置

#### 8.2.1 告警规则示例
```json
{
```

## 9. 最佳实践

### 9.1 查询优化

✅ **只查询需要的字段**
```json
❌ 不推荐: {"User": {"columns": []}}  // 查询所有字段
✅ 推荐: {"User": {"columns": ["id", "username"]}}  // 只查需要的字段
```

✅ **使用索引字段作为条件**
```json
❌ 不推荐: {"User": {"where": {"email": {"$like": "%@example.com"}}}
✅ 推荐: {"User": {"where": {"id": {"$in": [1, 2, 3]}}}  // 使用主键
```

✅ **限制返回数据量**
```json
❌ 不推荐: {"User": {"limit": 10000}}  // 一次返回太多数据
✅ 推荐: {"User": {"limit": 100}}  // 分页查询
```

✅ **使用缓存**
```json
✅ 推荐: {
  "User": {},
  "@cache": 300000  // 缓存5分钟
}
```

### 9.2 安全实践

✅ **使用参数化查询**
- 避免SQL注入
- 使用预编译语句

✅ **最小权限原则**
- 只授予必要的权限
- 定期审查权限

✅ **敏感数据保护**
- 密码字段不返回
- 个人信息脱敏显示

✅ **审计日志**
- 记录所有数据访问
- 记录权限变更

### 9.3 工作流设计

- 每个节点只做一件事

✅ **错误处理**

- 重复执行不会产生副作用

✅ **超时控制**
- 每个节点设置超时
- 避免无限等待

### 9.4 数据源管理

## 10. 常见问题

### 10.1 查询相关

**Q: 查询超时怎么办？**
A: 
1. 检查查询是否使用了索引
2. 减少返回的数据量
3. 使用`@cache`指令缓存结果
4. 联系管理员优化数据库

**Q: 如何查询大量数据？**
A: 
1. 使用分页：`{"limit": 100, "offset": 0}`
2. 使用流式返回：`"stream": true`
3. 导出为文件：选择CSV或Excel格式

**Q: JOIN查询很慢？**
A: 
1. 检查关联字段是否有索引
2. 减少JOIN的表数量
3. 考虑使用子查询替代复杂JOIN

### 10.2 数据源相关

**Q: 数据源连接失败？**
A:
1. 检查网络连通性
2. 验证用户名密码
3. 检查数据库是否允许远程连接
4. 查看防火墙设置

**Q: 如何切换数据源？**
A:
1. 在查询构建器中选择目标数据源
2. 系统自动使用对应的数据源执行查询
3. 可以保存多个数据源配置

**Q: 数据源密码如何加密？**
A:
1. 系统使用AES-256加密存储密码
2. 连接时自动解密
3. 密码只在内存中使用，不写入日志

### 10.3 权限相关

**Q: 没有权限访问某个表？**
A:
1. 联系管理员申请权限
2. 查看自己拥有的权限列表
3. 确认是否选择了正确的数据源

**Q: 如何申请更多权限？**
A:
1. 在权限管理页面提交申请
2. 填写申请理由和使用场景
3. 等待管理员审批

**Q: 权限变更后不生效？**
A:
1. 退出登录重新登录
2. 清除浏览器缓存
3. 联系管理员确认权限已生效

### 10.4 工作流相关

**Q: 工作流执行失败？**
A:
1. 查看执行日志定位失败节点
2. 检查节点配置是否正确
3. 验证数据源连接是否正常
4. 修复后重新执行

**Q: 如何查看工作流执行历史？**
A:
1. 进入工作流详情页面
2. 查看"执行历史"标签
3. 点击某次执行查看详细日志

**Q: 工作流如何调试？**
A:
1. 使用"测试执行"功能
2. 在每个节点设置断点
3. 查看中间变量的值
4. 使用调试日志输出

### 10.5 监控相关

**Q: 如何配置告警？**
A:
1. 进入"告警规则"页面
2. 点击"新建告警规则"
3. 选择监控指标和阈值
4. 配置告警通道（邮件/短信/Webhook）
5. 保存并启用规则

**Q: 如何查看历史监控数据？**
A:
1. 在监控图表中点击"查看历史"
2. 选择时间范围
3. 导出监控数据
4. 对比不同时间段的数据

**Q: 如何自定义仪表板？**
A:
1. 点击"创建仪表板"
2. 选择要展示的指标
3. 配置图表布局
4. 设置刷新频率
5. 保存仪表板

## 11. 快速参考

### 11.1 APIJSON语法速查表

| 功能 | 语法 |
|--------|------|
| 查询所有 | `{"Table": {}}` |
| 指定字段 | `{"Table": {"columns": ["field1", "field2"]}}` |
| WHERE条件 | `{"Table": {"where": {"field": "value"}}` |
| 大于 | `{"Table": {"where": {"field": {"$gt": 10}}}}` |
| 小于 | `{"Table": {"where": {"field": {"$lt": 10}}}}` |
| 等于 | `{"Table": {"where": {"field": {"$eq": 10}}}}` |
| 不等于 | `{"Table": {"where": {"field": {"$ne": 10}}}}` |
| 在列表中 | `{"Table": {"where": {"field": {"$in": [1, 2, 3]}}}}` |
| 模糊查询 | `{"Table": {"where": {"field": {"$like": "%keyword%"}}}}` |
| JOIN | `{"Table": {"joins": [{"type": "INNER", "table": "Other", "on": "Table.id = Other.tableId"}]}}` |
| 分组 | `{"Table": {"groupBy": ["field1", "field2"]}}` |
| 排序 | `{"Table": {"orderBy": ["field+", "field-"]}}` |
| 分页 | `{"Table": {"limit": 20, "offset": 0}}` |
| 缓存 | `{"Table": {}, "@cache": 300000}` |
